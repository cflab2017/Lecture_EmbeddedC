# 02. 자료형·진수·stdint·sizeof

PC에서 C를 짤 때는 `int`가 몇 바이트인지 거의 신경 쓰지 않습니다. 메모리가 넉넉하고, 어차피 컴파일러가 알아서 잘 처리하니까요. 하지만 임베디드는 다릅니다. MCU의 레지스터는 폭이 **정확히** 정해져 있고(8비트, 16비트, 32비트), RAM은 수 KB뿐입니다. "이 변수가 정확히 몇 비트인가"가 곧 "레지스터에 제대로 들어가는가", "메모리를 낭비하지 않는가"를 결정합니다.

이번 편은 보드 없이 순수 C 로직으로, **자료형의 실체(폭)** 를 들여다봅니다. `stdint.h`의 폭 고정 타입이 왜 임베디드의 사실상 표준인지, 같은 수를 10진·16진·2진으로 보는 법, `sizeof`로 크기를 확인하는 법, 그리고 부호 없는(unsigned) 값이 최댓값을 넘을 때 일어나는 **랩어라운드**까지 PC의 `gcc`로 직접 확인합니다.

## 학습 목표
- C 기본 자료형의 크기가 컴파일러·칩마다 다를 수 있음을 이해하고, 8051과 ARM/PC의 `int` 폭 차이를 설명한다.
- `stdint.h`의 폭 고정 타입(`uint8_t`/`uint16_t`/`uint32_t` 등)을 쓰는 이유를 레지스터 폭과 연결해 설명한다.
- 같은 수를 10진·16진(`0x`)·8진·2진으로 표기하고 `printf` 포맷(`%u %x %X %o`)으로 출력한다.
- `sizeof`로 자료형·변수의 바이트 크기를 확인한다.
- 부호 없는 자료형의 랩어라운드(예: `uint8_t` 255+1=0)와 같은 비트의 부호 해석 차이를 이해한다.

## 대상 환경
- 컴파일러: Keil C51(8051) / Keil MDK-ARM(STM32) — 이번 편은 순수 로직이라 둘 중 무엇이든, 또는 PC만으로도 충분.
- 디바이스: 순수 로직 — **PC + gcc 또는 μVision 시뮬레이터** (실제 보드 불필요).
- 검증: PC의 `gcc`(`main.c`, `pc_test.c`), 또는 μVision 시뮬레이터의 Watch 창.

## 핵심 개념

### 1) C 기본 자료형과 "크기가 칩마다 다르다"는 문제
C 표준은 각 자료형의 **최소 크기**만 보장하고, 정확한 바이트 수는 컴파일러와 대상 칩이 정합니다. 그래서 `int` 하나도 환경에 따라 폭이 다릅니다.

| 자료형 | 8051 (Keil C51) | ARM Cortex-M (MDK) | PC (x86-64 gcc) |
|--------|-----------------|--------------------|-----------------|
| `char` | 1 byte | 1 byte | 1 byte |
| `int` | **2 byte (16비트)** | 4 byte (32비트) | 4 byte (32비트) |
| `long` | 4 byte | 4 byte | 8 byte |
| 포인터 | 1~3 byte(메모리 영역별) | 4 byte | 8 byte |

`int`가 8051에서는 16비트, ARM에서는 32비트입니다. 8051에서 `int count;`로 50000까지 세려 했다면 16비트 한계(부호 있으면 -32768~32767)에 걸려 엉뚱한 값이 나옵니다. **"`int`니까 충분하겠지"** 라는 가정이 임베디드에서 깨지는 대표적인 지점입니다.

### 2) `stdint.h` — 폭이 이름에 박힌 자료형
해결책은 크기를 이름에 못 박은 타입을 쓰는 것입니다. `#include <stdint.h>`를 하면 다음을 쓸 수 있습니다.

| 타입 | 폭 | 범위 |
|------|----|------|
| `uint8_t` / `int8_t` | 8비트 | 0~255 / -128~127 |
| `uint16_t` / `int16_t` | 16비트 | 0~65535 / -32768~32767 |
| `uint32_t` / `int32_t` | 32비트 | 0~4294967295 / 약 ±21억 |

이름만 봐도 폭을 알 수 있어 **어떤 칩에서도 동일한 의미**를 가집니다. 게다가 레지스터는 폭이 정확히 정해져 있으므로 딱 맞는 타입을 고르면 의도가 분명해집니다.

- 8비트 포트/레지스터(예: 8051의 `P1`) → `uint8_t`
- 16비트 타이머 카운터 → `uint16_t`
- 32비트 STM32 주변장치 레지스터 → `uint32_t`

> 메모리도 절약됩니다. 0~200 범위 카운터에 `int`(8051에서 2바이트) 대신 `uint8_t`(1바이트)를 쓰면 RAM을 아낍니다. 수 KB뿐인 칩에서는 이 차이가 쌓입니다.

### 3) 진수 표기 — 10진·16진·8진·2진
레지스터와 비트를 다룰 때는 16진수가 자연스럽습니다. 16진 한 자리가 정확히 4비트(니블)에 대응하기 때문입니다.

| 표기 | 예 | 의미 |
|------|----|----|
| 10진 | `42` | 사람이 세는 수 |
| 16진 | `0x2A` | `0x` 접두, 한 자리 = 4비트 |
| 8진 | `052` | **`0` 접두** — 실수로 붙이면 8진이 됨(주의!) |
| 2진 | `0b00101010` | `0b`는 컴파일러 확장(C23부터 표준) — 이식성 위해 피함 |

`printf` 포맷 문자도 진수별로 다릅니다.

| 포맷 | 출력 | 예 (값 42) |
|------|------|-----------|
| `%u` | 부호 없는 10진 | `42` |
| `%d` | 부호 있는 10진 | `42` |
| `%x` / `%X` | 16진(소문자/대문자) | `2a` / `2A` |
| `%o` | 8진 | `52` |

표준 `printf`에는 2진 출력 포맷이 없습니다. 그래서 2진은 비트를 직접 뽑아 찍습니다: `(v >> i) & 1`로 `i`번째 비트를 0/1로 얻습니다(비트 연산은 3편에서 본격적으로).

### 4) `sizeof` — 자료형의 실체 확인
`sizeof`는 자료형이나 변수가 차지하는 **바이트 수**를 알려주는 연산자입니다(함수가 아닙니다).

```c
sizeof(uint16_t)   /* 2 */
sizeof(reg)        /* 변수 reg 의 크기 */
```

결과 타입은 `size_t`(부호 없는 정수)라서 `printf`로 찍을 때는 `(unsigned)`로 캐스팅하거나 `%zu`를 씁니다. 이번 예제는 C90 호환을 위해 `(unsigned)`로 캐스팅해 `%u`로 출력합니다. 칩이 바뀌면 `sizeof(int)`가 2가 될 수도, 4가 될 수도 있다는 것을 `sizeof`로 직접 확인할 수 있습니다.

### 5) 부호 없음/있음과 랩어라운드
부호 없는(unsigned) 타입은 음수를 표현하지 않습니다. 대신 최댓값을 넘으면 **0으로 되돌아옵니다(wrap-around).**

```c
uint8_t c = 255;
c = c + 1;   /* 256 이 아니라 0 */
```

이는 버그가 아니라 정의된 동작이며, 하드웨어 타이머 카운터가 도는 원리이기도 합니다. 한편 **같은 비트 패턴**도 부호 해석에 따라 값이 달라집니다. `0xFF`(8비트 모두 1)는 `uint8_t`로 읽으면 255, `int8_t`로 읽으면 -1(2의 보수)입니다. 레지스터 값을 부호 있는 변수에 잘못 담으면 음수로 둔갑하는 함정이 여기서 나옵니다.

## μVision 으로 보기 (이번 편은 PC로도 충분)
이번 편은 특정 디바이스에 묶이지 않은 순수 로직입니다. μVision에서 확인하려면 빈 프로젝트에 `main.c`를 넣고 디버그(Ctrl+F5)로 들어가, `reg` 같은 변수를 **Watch 창**에 추가한 뒤 우클릭 → `Number Base → Hexadecimal`로 진수를 바꿔 보며 같은 값이 다르게 보이는 것을 관찰할 수 있습니다. 정식 디바이스별 프로젝트 생성 절차는 6편(8051)·14편(STM32)에서 단계별로 안내합니다.

> 텍스트 출력 확인은 PC의 `gcc`가 훨씬 간단합니다. 아래 "PC에서 검증하기"로 바로 돌려 보세요.

## 예제로 보기

### 예제 1 — `main.c` : 자료형의 폭과 같은 값의 여러 진수
폭 고정 타입의 크기를 `sizeof`로 찍어 보고, 한 값(`0x2A`)을 10진·16진·8진·2진으로 나란히 출력합니다. 진수만 다를 뿐 같은 수임을 눈으로 확인하는 것이 목적입니다.

```c
#include <stdio.h>
#include <stdint.h>

/* 8비트 값을 2진수 8자리로 출력한다.
 * 표준 printf 에는 2진 출력 포맷이 없어 비트를 직접 뽑아 찍는다. */
static void print_bin8(uint8_t v)
{
    int i;
    for (i = 7; i >= 0; i--) {
        putchar(((v >> i) & 1u) ? '1' : '0');
    }
}

/*
 * 02강 예제 1 — 자료형의 실체(폭)와 같은 수의 여러 진수 표기를 본다.
 * stdint 의 폭 고정 타입은 어떤 칩(8051/STM32/PC)에서도 크기가 같다.
 */
int main(void)
{
    uint8_t reg = 0x2A;   /* 16진 0x2A = 10진 42 = 2진 00101010 */

    /* stdint 폭 고정 타입의 크기는 이름 그대로다 (어디서나 동일) */
    printf("sizeof(uint8_t)  = %u byte\n", (unsigned)sizeof(uint8_t));
    printf("sizeof(uint16_t) = %u byte\n", (unsigned)sizeof(uint16_t));
    printf("sizeof(uint32_t) = %u byte\n", (unsigned)sizeof(uint32_t));

    /* 같은 값 하나를 10진/16진/8진/2진으로 본다 */
    printf("\n");
    printf("dec = %u\n",     (unsigned)reg);
    printf("hex = 0x%02X\n", reg);
    printf("oct = 0%o\n",    reg);
    printf("bin = ");
    print_bin8(reg);
    printf("\n");

    return 0;
}
```

### 예제 2 — `pc_test.c` : 랩어라운드와 부호 해석
부호 없는 값이 최댓값을 넘을 때 0으로 도는 것, 그리고 같은 `0xFF` 비트가 부호 유무에 따라 255/-1로 갈리는 것을 확인합니다.

```c
#include <stdio.h>
#include <stdint.h>

/*
 * 02강 예제 2 — 부호 없음(unsigned) 자료형의 '랩어라운드' 와
 *               같은 비트를 부호 유무에 따라 다르게 해석하는 모습을 본다.
 * 레지스터·카운터에서 매일 만나는 현상이라 PC 에서 먼저 감을 잡는다.
 */
int main(void)
{
    uint8_t u    = 255;    /* uint8_t 최댓값 */
    uint8_t bits = 0xFF;   /* 동일한 8비트를 두 가지로 해석해 본다 */

    /* unsigned 오버플로는 '랩어라운드': 최댓값을 넘으면 0 으로 되돌아온다. */
    u = (uint8_t)(u + 1);
    printf("uint8_t 255 + 1 = %u\n", (unsigned)u);        /* 0 */

    /* 같은 0xFF 비트를 부호 없음 / 부호 있음으로 다르게 읽는다.
     * (int8_t)0xFF 는 2의 보수 환경(대부분의 MCU·PC)에서 -1 이다. */
    printf("0xFF as uint8_t = %u\n", (unsigned)bits);     /* 255 */
    printf("0xFF as int8_t  = %d\n", (int)(int8_t)bits);  /* -1  */

    return 0;
}
```

## PC에서 검증하기
순수 로직 편이므로 보드 없이 바로 확인할 수 있습니다.

```
gcc main.c -o main && ./main
```

예상 출력(stdout):

```
sizeof(uint8_t)  = 1 byte
sizeof(uint16_t) = 2 byte
sizeof(uint32_t) = 4 byte

dec = 42
hex = 0x2A
oct = 052
bin = 00101010
```

```
gcc pc_test.c -o pc_test && ./pc_test
```

예상 출력(stdout):

```
uint8_t 255 + 1 = 0
0xFF as uint8_t = 255
0xFF as int8_t  = -1
```

> 참고: `sizeof(int)`는 PC·ARM에서 4, 8051에서 2가 나옵니다. 위 예제가 `uint8_t`/`uint16_t`/`uint32_t`만 찍는 것은 이들이 **어디서나 1/2/4로 동일**해 결과가 흔들리지 않기 때문입니다. `int`의 폭이 환경마다 다르다는 것은 표(핵심 개념 1)에서 확인하세요.

## 시뮬레이터로 확인하기 (참고)
μVision 디버그 세션에서 `reg`를 Watch 창에 넣고, 우클릭 메뉴의 `Number Base`로 표기를 Decimal ↔ Hexadecimal로 바꿔 보면 같은 값이 `42` ↔ `0x2A`로 표시됩니다. `count++`를 F11로 한 줄씩 진행하며 `uint8_t` 변수가 255 다음에 0으로 도는 것도 눈으로 볼 수 있습니다. (`printf`의 시뮬레이터 출력은 UART 리타게팅 설정이 필요하므로, 텍스트 확인은 PC `gcc`가 간단합니다 — UART는 11·19편.)

## 자주 하는 실수

### Q. `int count;`로 5만까지 셌는데 값이 이상해요.
A. 8051(Keil C51)에서 `int`는 16비트라 부호 있으면 -32768~32767, 부호 없어도 0~65535입니다. 큰 수를 다룰 땐 폭을 못 박은 `uint32_t`(또는 `unsigned long`)를 쓰세요. "`int`면 충분"이라는 PC 습관이 16비트 칩에서 깨지는 대표 사례입니다.

### Q. `int x = 010;` 했더니 8이 아니라 10이 아니네요?
A. 숫자 앞에 `0`을 붙이면 **8진수**로 해석됩니다. `010`은 8진수라 10진 8입니다. 의도치 않은 선행 0은 버그의 원인이니, 10진수는 `0`으로 시작하지 말고, 16진수는 명시적으로 `0x`를 붙이세요.

### Q. `for (uint8_t i = 10; i >= 0; i--)`가 끝나지 않아요.
A. 부호 없는 `i`는 절대 음수가 되지 않습니다. `i`가 0일 때 `i--`하면 -1이 아니라 255로 랩어라운드되어 `i >= 0`이 영원히 참이 됩니다. 카운트다운에는 부호 있는 타입을 쓰거나, `while (i-- > 0)`처럼 조건을 다시 설계하세요.

### Q. 레지스터에서 읽은 0xFF를 변수에 담았더니 -1이 나와요.
A. 받는 변수가 `int8_t`/`signed char`라면 `0xFF`는 2의 보수로 -1이 됩니다. 레지스터 값처럼 "비트 묶음"은 부호 없는 타입(`uint8_t` 등)으로 받아야 의도대로 255로 읽힙니다.

## 정리
- C 기본 자료형의 크기는 칩마다 다르다. `int`는 8051에서 16비트, ARM/PC에서 32비트다.
- `stdint.h`의 `uint8_t`/`uint16_t`/`uint32_t`는 폭이 이름에 박혀 어디서나 동일하다 — 임베디드의 사실상 표준.
- 레지스터·비트는 16진수(`0x`)가 자연스럽고, 선행 `0`(8진)과 폭 가정에 주의한다.
- `sizeof`로 자료형의 실제 바이트 수를 확인할 수 있다.
- 부호 없는 값은 최댓값을 넘으면 0으로 랩어라운드되며, 같은 비트도 부호 해석에 따라 값이 달라진다.
