# 01. 임베디드 C와 Keil μVision 시작하기

PC에서 도는 C 프로그램과 마이크로컨트롤러(MCU)에서 도는 C 프로그램은 같은 언어를 쓰지만 사고방식이 다릅니다. 운영체제도, 화면도, 키보드도 없는 작은 칩 위에서 코드가 직접 하드웨어를 다뤄야 하죠. 이 강의는 그 첫걸음으로, **임베디드 C가 무엇인지**, 그리고 우리가 24편 내내 사용할 도구인 **Keil μVision**이 어떤 흐름으로 코드를 빌드·디버그·시뮬레이션하는지를 익힙니다.

아직 실제 보드(8051, STM32)는 다루지 않습니다. 대신 어디서나 통하는 순수 C 로직을 작성해 보고, μVision의 작업 흐름과 시뮬레이터 개념을 잡는 데 집중합니다. 보드가 없어도 끝까지 따라올 수 있도록, 모든 예제는 PC의 `gcc`로도 확인할 수 있게 준비했습니다.

## 학습 목표
- 임베디드 시스템과 임베디드 C의 특징을 일반 PC 프로그래밍과 비교해 설명한다.
- Keil μVision의 두 갈래(C51 / MDK-ARM)와 각각이 어떤 칩을 겨냥하는지 구분한다.
- 임베디드 프로그램의 표준 골격(`main` + `while(1)` 무한 루프)을 이해한다.
- μVision에서 프로젝트 생성 → 빌드(F7) → 디버그/시뮬레이션(Ctrl+F5)의 흐름을 따라간다.
- 순수 로직 예제를 `gcc`로 빌드·실행해 동작을 확인한다.

## 대상 환경
- 컴파일러: Keil C51(8051용) 또는 Keil MDK-ARM / Arm Compiler 6(STM32용) — 이번 편은 둘 다 설치만 확인.
- 디바이스: 순수 로직 — **PC + gcc 또는 μVision 시뮬레이터** (실제 보드 불필요).
- 검증: μVision 시뮬레이터, 그리고 PC의 `gcc`(`pc_test.c`).

## 핵심 개념

### 1) 임베디드 시스템과 임베디드 C
임베디드 시스템은 특정 기능을 수행하도록 기계 안에 내장된 컴퓨터입니다. 전자레인지, 세탁기, 드론, 자동차 ECU 안의 작은 칩(MCU)이 모두 그렇습니다. PC 프로그래밍과 비교하면 차이가 분명합니다.

| 구분 | PC 프로그램 | 임베디드 프로그램 |
|------|-------------|-------------------|
| 운영체제 | 있음(Windows/Linux) | 없는 경우가 많음(베어메탈) |
| 프로그램 종료 | `main`이 return하면 끝 | 끝나지 않고 `while(1)`로 계속 동작 |
| 입출력 | 화면·키보드·파일 | GPIO 핀, 센서, 레지스터 |
| 자원 | 수 GB RAM | 수 KB ~ 수백 KB RAM |
| 디버깅 | 콘솔 출력 | LED 토글, UART, 디버거 |

임베디드 C는 표준 C와 같은 문법을 쓰지만, **메모리 주소를 직접 다루고**, **하드웨어 레지스터를 비트 단위로 조작**하며, **자원이 매우 제한적**이라는 점이 다릅니다.

### 2) Keil μVision — C51 vs MDK-ARM
Keil μVision은 임베디드 개발용 통합 개발 환경(IDE)입니다. 코드 편집, 컴파일, 디버깅, 시뮬레이션을 한 화면에서 합니다. 어떤 칩을 쓰느냐에 따라 내부 컴파일러가 갈립니다.

| 도구 | 대상 칩 | 컴파일러 | 이 트랙에서 다루는 편 |
|------|---------|----------|----------------------|
| Keil C51 | 8051 계열(예: AT89C52) | C51 | 6~12편 |
| Keil MDK-ARM | Arm Cortex-M(예: STM32F103) | Arm Compiler 6 | 13~20, 24편 |

> 설치 시 둘 다 같은 μVision IDE를 공유하지만, 8051 디바이스 팩과 STM32 디바이스 팩을 각각 설치해야 해당 칩을 고를 수 있습니다.

### 3) 임베디드 프로그램의 표준 골격
PC 프로그램은 `main`이 끝나면 종료하지만, 임베디드 프로그램은 전원이 켜진 동안 계속 돌아야 합니다. 그래서 거의 항상 무한 루프가 들어갑니다.

```c
int main(void)
{
    /* 1. 초기화: 클럭, 핀, 주변장치 설정 (단 한 번) */

    while (1) {
        /* 2. 본문: 센서 읽고, 판단하고, 출력하고 ... 무한 반복 */
    }
    /* 여기에는 절대 도달하지 않는다 */
}
```

`main` 앞에는 우리가 직접 짜지 않는 **시작 코드(startup code)** 가 있습니다. 스택을 세팅하고 전역 변수를 초기화한 뒤 `main`으로 점프하죠. Keil은 디바이스 팩에 이 startup 파일을 포함하고 있어, 우리는 `main`부터 작성하면 됩니다.

### 4) 빌드 → 디버그 → 시뮬레이션 흐름
μVision의 핵심 단축키는 두 개입니다.

- **F7 (Build / Rebuild)**: 소스를 컴파일·링크해 실행 이미지를 만든다.
- **Ctrl+F5 (Start/Stop Debug Session)**: 디버거를 시작한다. 시뮬레이터 모드면 실제 보드 없이 칩 동작을 흉내 낸다.

디버그 세션에 들어가면 **F11(한 줄씩 실행)**, **F5(연속 실행)** 으로 코드를 따라가며 레지스터·변수 값을 눈으로 확인할 수 있습니다.

### 5) "시뮬레이터로 검증한다"는 것
이 트랙은 실제 하드웨어 없이 **μVision 내장 시뮬레이터**로 결과를 확인합니다. 시뮬레이터는 CPU 명령과 주변장치 동작을 소프트웨어로 흉내 냅니다. 그래서 결과는 눈에 보이는 형태여야 합니다.

- LED 예제 → Logic Analyzer 창에서 핀 토글 관찰
- UART 예제 → Serial 창에서 출력 텍스트 확인
- 타이머 예제 → 레지스터 창에서 카운터 값 증가 관찰

이번 편은 순수 로직이라, 보드 대신 **PC의 `gcc` + `printf`** 로 같은 로직을 바로 확인합니다.

## μVision 시작하기 (이번 편은 설치·둘러보기)
이번 편은 특정 디바이스에 묶이지 않으므로, 본격적인 디바이스별 프로젝트 생성은 6편(8051), 14편(STM32)에서 단계별로 안내합니다. 여기서는 도구를 준비하고 화면에 익숙해지는 것이 목표입니다.

### 컴파일러·IDE 다운로드 경로
μVision IDE는 아래 컴파일러 패키지에 함께 들어 있습니다. 쓰려는 칩에 맞는 것을 받으세요.

| 도구 | 용도(다루는 편) | 다운로드 경로 |
|------|-----------------|---------------|
| Keil MDK-Arm | STM32 등 Arm Cortex-M (13~20, 24편) | https://www.keil.com/demo/eval/arm.htm |
| Keil C51 | 8051 계열, 예: AT89C52 (6~12편) | https://www.keil.com/demo/eval/c51.htm |
| 전체 제품 목록 | MDK / C51 / C251 / C166 한 페이지 | https://www.keil.com/download/product/ |
| MDK 커뮤니티 에디션(무료) | 비상업용 무료 라이선스 활성화 안내 | https://www.keil.arm.com/mdk-community/ |

> - 다운로드 페이지는 간단한 등록 양식(이름·이메일 등)을 거친 뒤 설치 파일을 내려받습니다.
> - 라이선스 없이도 **Lite/Evaluation 에디션**으로 이 트랙의 예제는 모두 빌드·시뮬레이션할 수 있습니다(코드 크기 제한이 있으나 입문 예제 규모에는 충분). 더 큰 코드가 필요하면 위 **MDK 커뮤니티 에디션**(비상업용 무료)을 활성화하세요.
> - μVision은 **윈도우 전용**입니다. macOS/Linux라면 가상머신을 쓰거나, 1~5편은 PC의 `gcc`로 따라오세요.

### 설치 후 할 일
1. 위 표에서 **MDK-Arm**(STM32용)·**C51**(8051용)을 내려받아 설치한다. (둘 다 같은 μVision IDE를 공유)
2. μVision을 실행하고 **Pack Installer**에서 8051용 팩과 STM32F1 시리즈 팩을 설치한다.
3. 메뉴를 둘러본다: `Project`(프로젝트 생성), `Debug`(디버그 시작/정지), `View → Serial Windows`(UART 출력), `View → Analysis Windows → Logic Analyzer`(핀 파형).
4. 빈 프로젝트 하나를 만들어 보고 F7로 빌드가 도는지만 확인한다. (구체적 절차는 6·14편)

## 예제로 보기

### 예제 1 — `main.c` : 임베디드 골격을 PC에서 흉내내기
실제 보드라면 `led` 변수 대신 GPIO 핀이 LED를 켜고 끕니다. 여기서는 핀 대신 변수를 쓰고, 토글 결과를 `printf`로 출력해 "무한 루프 안에서 상태가 바뀐다"는 감각을 잡습니다. 무한 루프는 PC에서 관찰하기 어려우니 5회로 제한했습니다.

```c
#include <stdio.h>
#include <stdint.h>

/*
 * 01강 예제 — 임베디드 프로그램의 기본 골격을 PC에서 흉내낸다.
 * 실제 보드에서는 led 변수가 GPIO 핀(LED)에 연결되지만,
 * 여기서는 변수로 시뮬레이션하고 printf 로 상태를 출력한다.
 */
int main(void)
{
    uint8_t led = 0;   /* 0 = 꺼짐, 1 = 켜짐 */
    uint8_t tick;

    printf("Hello, Embedded C!\n");

    /*
     * 임베디드 main 은 보통 while(1) 무한 루프로 동작한다.
     * PC에서는 끝없이 돌면 관찰이 어려우므로 5회로 제한했다.
     */
    for (tick = 0; tick < 5; tick++) {
        led = (uint8_t)!led;                 /* LED 토글: 0<->1 */
        printf("tick %u: LED = %s\n",
               (unsigned)tick, led ? "ON" : "OFF");
    }

    return 0;
}
```

### 예제 2 — `pc_test.c` : 순수 로직 함수로 분리해 검증
토글이라는 "결정"을 `toggle()` 함수로 떼어내면, 하드웨어 없이도 그 로직만 단위로 검증할 수 있습니다. 임베디드 코드를 짤 때 자주 쓰는 방법으로, 하드웨어 의존부와 순수 로직을 나누는 첫 연습입니다.

```c
#include <stdio.h>
#include <stdint.h>

/* 순수 로직: LED 상태를 뒤집는다. 하드웨어 의존 없음 → PC에서 검증 가능. */
static uint8_t toggle(uint8_t state)
{
    return (uint8_t)!state;
}

int main(void)
{
    uint8_t led = 0;
    uint8_t i;

    for (i = 0; i < 5; i++) {
        led = toggle(led);
        printf("tick %u: LED = %s\n", (unsigned)i, led ? "ON" : "OFF");
    }

    return 0;
}
```

## PC에서 검증하기
순수 로직 편이므로 보드 없이 바로 확인할 수 있습니다.

```
gcc pc_test.c -o pc_test && ./pc_test
```

예상 출력(stdout):

```
tick 0: LED = ON
tick 1: LED = OFF
tick 2: LED = ON
tick 3: LED = OFF
tick 4: LED = ON
```

`main.c`도 같은 방식으로 빌드하면 첫 줄에 `Hello, Embedded C!`가 더 붙어 출력됩니다.

```
gcc main.c -o main && ./main
```

```
Hello, Embedded C!
tick 0: LED = ON
tick 1: LED = OFF
tick 2: LED = ON
tick 3: LED = OFF
tick 4: LED = ON
```

## 시뮬레이터로 확인하기 (참고)
μVision에서 빈 프로젝트에 `main.c`를 넣고 디버그(Ctrl+F5)로 들어간 뒤, `led`와 `tick`을 Watch 창에 추가해 F11로 한 줄씩 실행하면 값이 0↔1로 바뀌는 것을 관찰할 수 있습니다. (단, 이 예제의 `printf`는 시뮬레이터에서 UART 리타게팅 설정이 필요하므로, 텍스트 출력 확인은 PC `gcc` 쪽이 훨씬 간단합니다. UART 출력은 11·19편에서 정식으로 다룹니다.)

## 자주 하는 실수

### Q. 임베디드 `main`은 왜 `return`하지 않나요? `return 0;`을 적었는데요.
A. PC에서는 `main`이 끝나면 OS가 프로세스를 정리합니다. 하지만 베어메탈 MCU에는 돌아갈 OS가 없습니다. `main`이 return하면 갈 곳이 없어 정의되지 않은 동작에 빠지죠. 그래서 실제 임베디드 `main`은 `while(1)`로 끝나며 절대 반환하지 않습니다. 이번 예제는 PC 관찰용이라 일부러 종료시킨 것입니다.

### Q. `int`를 쓰면 되는데 왜 `uint8_t` 같은 걸 쓰나요?
A. MCU마다 `int` 크기가 다를 수 있고(8051에서는 16비트), 레지스터는 폭이 정확히 정해져 있습니다. `uint8_t`/`uint16_t`/`uint32_t`는 폭이 이름에 박혀 있어 이식성과 의도가 분명합니다. 자세한 자료형 이야기는 2편에서 다룹니다.

### Q. `gcc`로 잘 되는데, 이게 정말 임베디드 학습이 맞나요?
A. 이번 편의 로직(토글, 무한 루프, 자료형)은 하드웨어와 무관한 C의 토대입니다. 토대를 PC에서 빠르게 다진 뒤, 6편부터 실제 칩의 레지스터로 같은 개념을 적용합니다. "순수 로직은 PC로, 하드웨어 제어는 시뮬레이터로"가 이 트랙의 방식입니다.

### Q. μVision 설치가 너무 무겁고 복잡해요.
A. 지금 당장 설치하지 않아도 됩니다. 1~5편(기초)은 모두 `gcc`로 따라올 수 있습니다. 8051 실습이 시작되는 6편 전까지 여유 있게 설치하세요.

## 정리
- 임베디드 C는 OS·풍부한 자원에 기대지 않고 하드웨어를 직접 다루는 C이다.
- Keil μVision은 C51(8051)과 MDK-ARM(STM32)으로 갈리며, 같은 IDE를 공유한다.
- 임베디드 프로그램의 골격은 `초기화 + while(1) 무한 루프`이며 `main`은 반환하지 않는다.
- μVision 흐름은 F7(빌드) → Ctrl+F5(디버그/시뮬) → 창으로 결과 관찰이다.
- 순수 로직은 `gcc`로 빠르게 검증하고, 하드웨어 제어는 시뮬레이터로 확인한다.
